Developer Onboarding Guide

Welcome to the engineering team! This guide walks you through everything you need to be productive in your first week. Follow the steps in order — each section builds on the previous one. If you get stuck, ping the #dev-onboarding Slack channel.

Account Setup

Before you can access any system, IT needs to provision your accounts. Submit a ticket to IT via the Service Desk portal within your first hour. Include your full name, start date, team (Engineering), and manager's name.

Required Accounts

  • GitHub Enterprise — Request access to the acme-corp organisation. You need to be added to the engineering team by your manager. Two-factor authentication is mandatory; use an authenticator app, not SMS.
  • JIRA — Your manager will add you to the relevant project boards. Default role is Developer. If you need admin access for a project, raise a request.
  • Confluence — Access is granted automatically once your corporate account is created. You can view all spaces but only edit spaces you are a member of.
  • Slack — Download the desktop app and the mobile app. Join: #general, #engineering, #dev-onboarding, #incidents, and your team channel (ask your manager).
  • Datadog — Request read-only access via IT. If you need to create monitors or dashboards, request the Developer role.
  • AWS Console — Read-only access to the dev and staging accounts is granted by default. Production access requires a separate approval from your VP Engineering.

Development Environment Setup

We standardise on macOS for development. If you received a Windows machine, raise a hardware swap request with IT immediately — our tooling is not tested on Windows.

Required Tools

Install the following in order. Each tool has a minimum version requirement enforced by our CI pipeline — installing an older version will cause build failures.

  • Homebrew — Package manager. Install from brew.sh. All other tools are installed via Homebrew.
  • Java 17 (Temurin) — brew install --cask temurin17. We use Temurin (Eclipse Adoptium), not Oracle JDK. Set JAVA_HOME in your shell profile.
  • Maven 3.9+ — brew install maven. Run mvn -version to verify.
  • Docker Desktop — Download from docker.com. Required for running local services. Allocate at least 8 GB RAM and 4 CPUs in Docker Desktop preferences.
  • kubectl — brew install kubectl. Configure with our cluster credentials (see Kubernetes Access guide).
  • IntelliJ IDEA Ultimate — Licences are managed via IT. Request a licence seat before installing. Community edition is not recommended as it lacks Spring Boot support.
  • AWS CLI v2 — brew install awscli. Configure with your SSO profile: aws configure sso.

Shell Configuration

Add the following to your ~/.zshrc (or ~/.bashrc if you use bash):

export JAVA_HOME=$(/usr/libexec/java_home -v 17)
export PATH="$JAVA_HOME/bin:$PATH"
export MAVEN_OPTS="-Xmx2g -XX:+TieredCompilation"

Run source ~/.zshrc after saving. Verify with java -version — it should report OpenJDK 17.

Repository Access and Structure

All source code is on GitHub Enterprise at github.acme.com. Clone repositories using SSH, not HTTPS — HTTPS tokens expire and break CI. Add your SSH public key to your GitHub profile under Settings → SSH and GPG keys.

Core Repositories

  • acme/MyApp — Main backend service (Spring Boot, Java). This is the primary repo most developers work in.
  • acme/frontend — React frontend. Separate repo with its own CI pipeline.
  • acme/infra — Terraform infrastructure code. Read-only for most developers; write access requires infrastructure team approval.
  • acme/shared-libs — Internal Java libraries published to our Nexus artifact server. Changes here affect all downstream services.
  • acme/jenkins-pipelines — Shared Jenkins pipeline library. Jenkinsfiles in application repos reference this.

Branch Strategy

We use trunk-based development with short-lived feature branches. Branch naming convention: feature/JIRA-ticket-short-description or fix/JIRA-ticket-description. Branch lifetimes should not exceed 2 days. Long-running branches lead to integration hell.

Protected branches: main and release/*. Direct pushes are blocked. All changes go through Pull Requests with at least 2 approvals.

Local Development Workflow

Before running the application locally, you need to start the dependent services using Docker Compose. The compose file is in the repository root.

# Start all dependencies (PostgreSQL, Redis, Kafka, localstack)
docker compose up -d

# Build the application
mvn clean package -DskipTests

# Run with local profile
mvn spring-boot:run -Dspring-boot.run.profiles=local

The application starts on port 8080. The health endpoint is at http://localhost:8080/actuator/health. If it returns 200, you are good.

Running Tests

Unit tests: mvn test — runs in approximately 2 minutes. Integration tests: mvn verify -P integration-tests — requires Docker Compose services to be running; takes 8–12 minutes. Do not skip unit tests when submitting a PR.

First Week Checklist

Use this checklist to track your onboarding progress. Share your completed checklist with your manager at your end-of-week 1:1.

  • IT accounts provisioned (GitHub, JIRA, Confluence, Slack, Datadog, AWS)
  • Dev environment set up (Java 17, Maven, Docker, IntelliJ)
  • SSH key added to GitHub
  • Cloned acme/MyApp and built successfully
  • Run unit tests locally — all passing
  • Read the Architecture Decision Records (ADRs) in Confluence → Engineering → ADRs
  • Joined all required Slack channels
  • Read the Code Review Guidelines (this wiki)
  • Read the Production Deployment Process (this wiki)
  • Attended team standup (daily, 9:30 AM)
  • 1:1 with manager scheduled
  • First JIRA ticket assigned and in progress
