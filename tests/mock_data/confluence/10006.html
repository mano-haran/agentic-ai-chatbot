Code Review and Pull Request Guidelines

Code review is one of the most valuable practices in our engineering culture. It improves code quality, spreads knowledge, and catches bugs before they reach users. These guidelines ensure reviews are thorough, constructive, and consistent across teams.

Creating a Pull Request

A good PR makes the reviewer's job easy. Before opening a PR, ask yourself: "Could a team member who wasn't involved in this work understand what changed and why from the PR description alone?" If the answer is no, improve the description.

PR Size

Keep PRs small and focused. A PR should do one thing. If you are fixing a bug and you noticed a refactoring opportunity, do them in separate PRs. The ideal PR size is 200–400 lines of production code change (excluding tests and generated code). PRs over 1000 lines must be justified with a comment — consider whether they can be split.

Large PRs are harder to review, take longer to merge, and cause more merge conflicts. If your feature is large, use feature flags and merge incrementally behind the flag.

PR Title

Format: [JIRA-TICKET] Short description of what changed. Example: [MYAPP-1234] Add email validation to user registration. Do not use vague titles like "Fix bug" or "Updates".

PR Description Template

Use the PR template that appears automatically when you open a PR on GitHub. Fill in every section:

  • What changed: A plain English summary of the changes. What does this code do differently from before?
  • Why: The business or technical reason for the change. Link the JIRA ticket.
  • How to test: Steps a reviewer can follow to manually verify the change. Include test data or environment setup if needed.
  • Risks: Any areas of uncertainty, potential side effects, or changes to behaviour that reviewers should pay particular attention to.
  • Screenshots: For UI changes, before and after screenshots are mandatory.

Review Standards

As a reviewer, your responsibility is to ensure the code is correct, maintainable, secure, and consistent with the codebase. Be thorough — a missed bug in review will become a production incident or a customer complaint.

What to Review

  • Correctness: Does the code do what the PR description says? Are there edge cases that are not handled (null inputs, empty lists, concurrent access, network timeouts)?
  • Tests: Are there adequate tests for the new code? Do the tests test behaviour, not implementation? Are happy paths, sad paths, and edge cases covered?
  • Security: Is user input validated and sanitised? Are there SQL injection, XSS, or path traversal vulnerabilities? Are secrets handled correctly (not logged, not in version control)?
  • Performance: Are there N+1 database queries? Are large results paginated? Are expensive operations cached?
  • Maintainability: Is the code readable? Are variable and method names clear? Is there duplication that should be extracted? Are there comments for non-obvious logic?
  • Backwards compatibility: Does the change break any existing API contracts? If a database column is renamed, is the migration handled correctly?

Comment Conventions

Use the following prefixes to make your intent clear:

  • nit: Minor style issue. Non-blocking. Author can fix or ignore at their discretion.
  • question: Clarification request. Not blocking if author provides a satisfying explanation.
  • suggestion: Alternative approach to consider. Non-blocking.
  • blocker: Must be addressed before merge. Explicitly label blockers so authors know.

Write comments as feedback on the code, not the author. "This method does X which could cause Y in Z scenario" is better than "Why did you write it this way?" Give specific suggestions when raising an issue — "Consider using Optional.orElseThrow() here to make the null handling explicit" is more helpful than "This could throw a NullPointerException."

Approval Requirements

All PRs require at least 2 approvals before merge. Rules:

  • At least 1 approval must come from a developer outside your immediate pair (you cannot both approve each other's changes).
  • For changes to core infrastructure code (framework, shared libraries, CI/CD), at least 1 approval must come from a Senior Engineer or above.
  • For changes to security-sensitive code (authentication, authorisation, payment processing, PII handling), a security review from the AppSec team is required in addition to the 2 engineer approvals.
  • Approvals are invalidated by new commits after the approval. A reviewer must re-approve after any non-trivial change.

Merge Policy

We use squash merges for all PRs. Each merged PR appears as a single commit on the main branch. Squash merge keeps the main branch history clean and readable.

The merge commit message is the PR title plus the PR number. Ensure your PR title accurately describes the final state of the change before merging, not just the work you started with.

Do not merge your own PRs even if you have the required approvals. The second reviewer merges, or the PR author asks the reviewing team lead to merge. This ensures a final human review at merge time rather than approving and forgetting.

Pre-Merge Checks

GitHub branch protection rules enforce: at least 2 approvals, CI pipeline passing (all tests green), no unresolved review threads, and branch up to date with main. You cannot override these — if any check is red, you cannot merge. Resolve all review comments before requesting re-review.

Code Review Checklist for Reviewers

Use this checklist when reviewing. You do not need to comment on every item — only raise issues when you find a problem. But do check every item.

  • PR description is complete and explains the "what" and "why"
  • Changes match the PR description and linked JIRA ticket
  • All new code has test coverage
  • Tests are meaningful (not just line coverage for coverage's sake)
  • No hardcoded secrets, credentials, or environment-specific values
  • Error cases are handled with appropriate logging
  • Database queries use parameterised statements
  • API responses do not leak sensitive data
  • No System.out.println or console.log in production code
  • New dependencies are justified and vetted for security vulnerabilities
  • Logging is at appropriate levels (not DEBUG in hot paths)
  • Breaking changes are documented and migration path is provided

Review Turnaround Time

Reviews should be completed within 1 business day of being requested. If you cannot review within that time, comment on the PR to let the author know and suggest an alternative reviewer. Blocked PRs stall the team — do not leave a PR waiting more than 2 business days without communication.

If your PR has been waiting for review for more than 1 business day, ping the reviewer in Slack. After 2 business days with no response, ask your team lead to assign a different reviewer.
