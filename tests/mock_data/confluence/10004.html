Kubernetes Cluster Access

We run three Kubernetes clusters: dev, staging, and production. This page explains how to get access, configure kubectl, and use the clusters safely. Production cluster access is restricted — read the access controls section carefully before requesting access.

Requesting Access

Submit an access request via the IT Service Desk with the following information: your name, team, manager, the cluster(s) you need access to, and the justification. Standard processing time is 1 business day for dev/staging and 3 business days for production.

Access levels available:

  • read-only — View pods, logs, events, services, configmaps. Cannot modify anything. Default for all developers.
  • developer — read-only plus: port-forward, exec into pods, scale deployments in your team's namespace. Standard for most developers.
  • namespace-admin — Full control within your team's namespace. For team leads. Requires manager and VP approval.
  • cluster-admin — Full cluster access. Platform team only. Not available to application developers under any circumstances.

Configuring kubectl

We use AWS EKS for all clusters. Authentication uses AWS SSO — you must complete AWS CLI setup first (see Developer Onboarding Guide). Once you have AWS access, fetch the kubeconfig for each cluster.

Getting the Kubeconfig

# Dev cluster
aws eks update-kubeconfig --name acme-dev --region eu-west-1 --profile acme-dev

# Staging cluster
aws eks update-kubeconfig --name acme-staging --region eu-west-1 --profile acme-staging

# Production cluster (requires production AWS access)
aws eks update-kubeconfig --name acme-prod --region eu-west-1 --profile acme-prod

After running these commands, your ~/.kube/config will have three contexts. Verify with kubectl config get-contexts.

Switching Between Clusters

Always verify which cluster you are connected to before running any command, especially before any mutating operation.

# See current context
kubectl config current-context

# Switch to dev
kubectl config use-context arn:aws:eks:eu-west-1:123456789:cluster/acme-dev

# Switch to staging
kubectl config use-context arn:aws:eks:eu-west-1:123456789:cluster/acme-staging

# Switch to production (use with extreme care)
kubectl config use-context arn:aws:eks:eu-west-1:123456789:cluster/acme-prod

Install kubectx for easier context switching: brew install kubectx. This gives you kubectx dev, kubectx staging shortcuts.

Namespaces and Team Organisation

Each team has its own namespace in each cluster. The naming convention is team-environment. For example, the backend team's namespaces are backend-dev, backend-staging, and backend-prod. Your RBAC role only grants access within your team's namespaces — you cannot read other teams' secrets or configmaps.

System namespaces (kube-system, monitoring, logging, ingress-nginx) are read-only for all developers. Contact the platform team for changes to these.

Common kubectl Commands

These are the commands you will use most often day-to-day.

Viewing Resources

# List pods in your namespace
kubectl get pods -n backend-dev

# Get pod details (including events and conditions)
kubectl describe pod myapp-6b7d9c8f4-xjk2p -n backend-dev

# Watch pod status in real time
kubectl get pods -n backend-dev -w

# Get resource usage (requires metrics-server)
kubectl top pods -n backend-dev

Viewing Logs

# Logs for a pod
kubectl logs myapp-6b7d9c8f4-xjk2p -n backend-dev

# Follow (stream) logs
kubectl logs -f myapp-6b7d9c8f4-xjk2p -n backend-dev

# Logs for all pods in a deployment (very useful)
kubectl logs -f deployment/myapp -n backend-dev

# Previous container logs (if pod crashed and restarted)
kubectl logs myapp-6b7d9c8f4-xjk2p -n backend-dev --previous

Port Forwarding

# Forward local port 8080 to the pod's port 8080
kubectl port-forward pod/myapp-6b7d9c8f4-xjk2p 8080:8080 -n backend-dev

# Forward to the service (load-balanced across all pods)
kubectl port-forward svc/myapp 8080:80 -n backend-dev

Exec into a Pod

# Open an interactive shell
kubectl exec -it myapp-6b7d9c8f4-xjk2p -n backend-dev -- /bin/bash

# Run a single command
kubectl exec myapp-6b7d9c8f4-xjk2p -n backend-dev -- curl -s localhost:8080/actuator/health

Debugging Pods

When a pod is not starting or is crash-looping, work through these steps in order.

Check Pod Status

Run kubectl get pod <name> -n <namespace>. The STATUS column tells you the current state. Common states:

  • Pending — Pod not scheduled yet. Usually means no node has enough resources, or a PVC is not bound.
  • ImagePullBackOff — Cannot pull the container image. Check that the image tag exists in the registry and the pull secret is configured.
  • CrashLoopBackOff — Container keeps crashing on startup. Check the logs with --previous flag to see the crash output.
  • OOMKilled — Out of memory. The container exceeded its memory limit. Increase the limit in the deployment manifest or find the memory leak.

Check Events

Kubernetes events contain detailed failure reasons that are not in the pod logs: kubectl describe pod <name> -n <namespace>. Scroll to the Events section at the bottom. Look for Warning events — they usually explain exactly what went wrong (e.g. "Insufficient memory", "Failed to pull image").

Check ConfigMaps and Secrets

Missing or misconfigured environment variables often cause startup crashes. Check what the pod sees: kubectl exec <pod> -n <ns> -- env | sort. Compare with the deployment's env section and the referenced configmaps and secrets.

Production Safety Rules

Violating these rules will result in access revocation regardless of role:

  • Never delete a production pod manually unless explicitly instructed during an incident by the on-call lead.
  • Never scale a production deployment to zero without a change ticket and approval.
  • Never exec into a production pod and modify files or state — use code deployments.
  • Do not read production secrets unless you have an approved need. Accessing secrets leaves an audit trail.
  • Do not run load tests or resource-intensive jobs against production clusters.
