Jenkins Setup and Configuration

This page covers how to configure Jenkins for your team's pipelines, manage credentials, use the shared pipeline library, and troubleshoot common build failures. Jenkins is our primary CI/CD platform, running at https://jenkins.acme.com.

Accessing Jenkins

Log in using your corporate SSO credentials at https://jenkins.acme.com. If this is your first time, you need to be added to the engineering organisation in Jenkins — ask your team lead to add you via Manage Jenkins → Manage and Assign Roles. New developers are given the Developer role by default, which allows triggering builds and viewing logs but not changing job configuration.

Jenkins Credentials Management

Credentials (API keys, passwords, SSH keys) are stored in the Jenkins credential store, never in source code or Jenkinsfiles. All secrets in source code will be caught by our git pre-commit hook and the commit will be blocked.

Adding New Credentials

Navigate to Jenkins → Manage Jenkins → Credentials → (global) → Add Credentials. Choose the correct credential type:

  • Username with password — For service accounts (e.g. Nexus, Docker registry)
  • Secret text — For API tokens and simple string secrets
  • SSH username with private key — For Git SSH authentication
  • Certificate — For PKCS12 keystores

Credential IDs follow the naming convention: service-environment-purpose. Examples: nexus-prod-deploy, github-token-ci, aws-staging-ecr. Use lowercase and hyphens only.

Using Credentials in Pipelines

Access credentials in your Jenkinsfile using the credentials() binding or the withCredentials block:

// Secret text or username+password
environment {
    NEXUS_CREDS = credentials('nexus-prod-deploy')
}

// Inside a stage
steps {
    withCredentials([usernamePassword(credentialsId: 'nexus-prod-deploy',
                                     usernameVariable: 'NEXUS_USER',
                                     passwordVariable: 'NEXUS_PASS')]) {
        sh 'mvn deploy -Dnexus.user=$NEXUS_USER -Dnexus.pass=$NEXUS_PASS'
    }
}

Never echo credentials to the console. Jenkins masks bound credential values in logs, but set -x in shell scripts can still expose them — avoid using set -x in pipeline steps that use credentials.

Rotating Credentials

When a credential needs to be rotated (e.g. token expiry, security incident), update the credential in Jenkins first, then verify pipelines still work before revoking the old token. Revoking before updating causes immediate build failures across all pipelines using that credential.

Pipeline Configuration

All pipelines are defined as Jenkinsfiles in the repository root. We use declarative pipeline syntax. Scripted pipeline syntax is not permitted for new pipelines (legacy pipelines are being migrated).

Standard Pipeline Structure

@Library('acme-shared-lib@main') _

pipeline {
    agent { label 'java-17' }

    options {
        timeout(time: 30, unit: 'MINUTES')
        buildDiscarder(logRotator(numToKeepStr: '20'))
        disableConcurrentBuilds()
    }

    environment {
        APP_NAME = 'myapp'
        DOCKER_REGISTRY = 'registry.acme.com'
    }

    stages {
        stage('Checkout') {
            steps {
                checkout scm
            }
        }
        stage('Build') {
            steps {
                sh 'mvn clean package -DskipTests -B'
            }
        }
        stage('Test') {
            steps {
                sh 'mvn test -B'
            }
            post {
                always {
                    junit 'target/surefire-reports/**/*.xml'
                }
            }
        }
        stage('Docker Build & Push') {
            when { branch 'main' }
            steps {
                script {
                    dockerBuildAndPush(APP_NAME, DOCKER_REGISTRY)
                }
            }
        }
    }

    post {
        failure {
            slackNotify(channel: '#builds', message: "FAILED: ${env.JOB_NAME} #${env.BUILD_NUMBER}")
        }
    }
}

Shared Pipeline Library

The shared library (acme/jenkins-pipelines on GitHub) contains reusable pipeline functions. Import it with @Library('acme-shared-lib@main') _ at the top of your Jenkinsfile.

Available Functions

  • dockerBuildAndPush(appName, registry) — Builds and pushes a Docker image tagged with the Git commit SHA and the branch name.
  • slackNotify(channel, message) — Sends a Slack notification. Picks up the webhook URL from the Jenkins credential slack-webhook-url.
  • runSmokeSuite(environment) — Runs the Maven smoke test profile against the specified environment.
  • kubeRollingDeploy(app, namespace, imageTag) — Performs a kubectl rolling update and waits for rollout completion.
  • sonarScan(projectKey) — Runs SonarQube analysis and enforces the quality gate.

Build Agents

Build agents are provisioned dynamically from Kubernetes. Each agent type has a specific label:

  • java-17 — Java 17 + Maven 3.9 + Docker CLI. Use for all Java builds.
  • node-20 — Node.js 20 + npm + Docker CLI. Use for frontend builds.
  • terraform — Terraform 1.7 + AWS CLI + kubectl. Use for infrastructure changes.
  • docker — Docker-in-Docker. Use only when you need to run containers inside the build.

Agents start in under 60 seconds. If your build is waiting more than 2 minutes for an agent, check the Kubernetes node capacity in the jenkins namespace: kubectl get nodes and kubectl top nodes.

Troubleshooting Common Build Failures

Compilation Errors

If you see [ERROR] COMPILATION ERROR in the build log, the Java source code cannot be compiled. Common causes: missing method in an interface (another developer changed the interface without updating all implementations), incompatible types (API change in a shared library), or import resolution failure (dependency not in pom.xml). Check the exact error line — it will reference the file and line number. Compare your branch's pom.xml dependencies with main's to spot version mismatches.

Test Failures

When tests fail, look for Tests run: X, Failures: Y in the Maven output. Click the test results link in Jenkins to see the full failure report. The most common causes are: mocking setup incorrect (verify your @MockBean annotations), database state not reset between tests (use @Transactional or @DirtiesContext), or a timing issue in async tests (increase timeouts or use Awaitility).

Docker Push Failures

If docker push fails with authentication error, the Jenkins credential aws-staging-ecr may have expired. ECR tokens expire every 12 hours. The shared library function dockerBuildAndPush refreshes the token automatically, but if you are not using the shared library, add aws ecr get-login-password | docker login ... before the push.

Out of Memory Errors

If Maven fails with OutOfMemoryError: Java heap space, the agent's memory limit is too low for your build. Set MAVEN_OPTS in your Jenkinsfile: environment { MAVEN_OPTS = '-Xmx2g' }. If already set and still failing, the agent pod may need a higher memory limit — contact the platform team.

Slow Builds

If builds are taking more than 20 minutes, first check if you can parallelize test stages. Use the Maven Surefire forkCount setting to run tests in parallel: add -DforkCount=2 -DreuseForks=true to your test command. If builds were fast before and have slowed recently, check if a new dependency is causing slow resolution — a Nexus proxy issue can cause Maven to wait for each dependency.
